About Me

I've always been curious about how things work. Coding stuck because it turned that curiosity into something I could actually build. Early on it was small wins: Excel formulas that ran a spreadsheet for me, CNC machines that cut what I'd programmed. When I started asking why those tools behaved the way they did, the bigger picture clicked. Code sits under almost everything now, and once you see that it's hard to unsee.

That pull is what pushed me into Computer Science at Manchester Metropolitan University. I'm starting final year in October and graduating in 2027. The projects on this site are me following the same habit: pick something interesting, open it up, and work out how to make it myself.

Where I want to end up is simple. I want a job where I build things that might spark the same reaction in someone else, that moment of “how does that work?” that made me fall into this. If someone tries coding because something I made got them curious, that's the dream.

Project Insights

NovaBank: online banking app

Why I Built It

I wanted a bigger project with a real database, auth, two user roles, and a frontend I could demo. A bank made sense because the hard parts are clear: money can't vanish between two accounts, and customers shouldn't see each other's statements. First time using TypeScript, Next.js, Prisma and PostgreSQL together.

What I Built

  • Register/login with bcrypt and JWT. New users automatically get a current account, savings account and debit card
  • Deposits, withdrawals and transfers by account number, all recorded with references like a real bank
  • Dashboard with balances, recent activity and a monthly spending chart (Chart.js)
  • Statement page with type and date filters, search, pagination and CSV export
  • Card freeze/unfreeze, notifications, dark mode
  • Admin panel with customer search, account freezing, every transaction in the bank, and an audit log
  • Rate limiting on the auth routes and login history tracking

What I Learned

The transfer logic was the bit I sweated over: taking money out of one account and putting it in another has to happen together or not at all, so both updates run inside one Prisma database transaction. I also learned to store money as decimals instead of floats, to never send full card numbers to the frontend, and to check ownership on every single route. Early on you could read anyone's statement just by changing the id in the URL, which is exactly the kind of bug that looks fine until someone tries it.

LuckyReels: online casino

Why I Built It

It started as a frontend-only slot machine: three reels, CSS animations, Web Audio sound effects. Fun, but there was a problem: the balance lived in the browser, so anyone could open devtools and give themselves money. That's the exact reason real betting sites never trust the client, so I rebuilt it the way they work: the browser just animates, and a Node server decides every outcome and owns every balance.

What I Built

  • Three games (slots, hi-lo cards, and a lucky wheel), all decided server-side with crypto-secure randomness
  • Accounts with bcrypt hashing, JWT in httpOnly cookies, and a 15-minute lockout after five failed logins
  • An 18+ date-of-birth check at registration, like UK casinos are required to have
  • A cashier: Luhn-validated cards (only the last 4 digits stored), deposits, withdrawals back to the card, and a full statement
  • Every penny recorded in a ledger. One atomic function moves all money, so the balance and statement can never disagree
  • Daily deposit limits enforced on the server
  • Rate limiting, whitelisted bet amounts, and Helmet security headers

What I Learned

Where trust sits in a client/server app. The browser asks to spin; the server debits the bet, rolls the symbols with crypto.randomInt, credits any winnings inside a SQLite transaction, and only then tells the frontend what to animate. I also worked out the expected value of each game: the wheel returns about 87p per £1 long-term, and learned to store money as integer pennies, because floats and money don't mix.

ChatWire: real-time chat

Why I Built It

Everything else I'd built was request/response: the browser asks, the server answers, done. Chat doesn't work that way. Messages have to appear for everyone in the room as they happen, without refreshing. I wanted to learn websockets properly, so I built a room chat with Flask-SocketIO and tested it with two browser tabs open like two different users.

What I Built

  • Join a named room with a display name
  • Live message send and receive across all clients in that room
  • System messages when someone joins or leaves
  • Flask-SocketIO backend with an HTML, CSS and JavaScript client
  • Separate handling for loading recent history vs appending new live messages

What I Learned

Socket events are not REST. If you treat every message like a normal HTTP response, the UI drifts: history loads twice, or a message shows in one tab and not the other. I had to keep room state clear on the server and make the client append only new events after the initial history load. Testing with two tabs open made those bugs obvious quickly.

ShopSphere: online shop

Why I Built It

VaultComics in first year stored the basket in the browser. I wanted the same shop idea, but with the cart living on the server so it survived a refresh and couldn't be edited from DevTools. Flask sessions were enough for that without standing up another database.

What I Built

  • Product catalogue with images, prices and search
  • Session cart: add items, change quantities, clear, and checkout
  • Server-side totals so the basket can't be faked in the browser
  • Flask backend with HTML, CSS and JavaScript frontend

What I Learned

Sessions feel simple until you hit edge cases: empty cart, quantity zero, checkout with nothing in it. Keeping the cart shape consistent on the server and re-rendering the basket page from that state was cleaner than trying to sync two copies of the data. It was also a useful step between a static shop and something with a real database.

StreamFlix: movie streaming app

Why I Built It

I wanted a full web project with a real backend, not just static pages. A movie browser with search, login and a watchlist was a clear brief: Node.js and Express on the back end, JWT for auth, and vanilla HTML/CSS/JavaScript on the front talking to the API over HTTP.

What I Built

  • Node.js and Express REST API with register/login
  • JWT auth so protected routes reject requests without a valid token
  • Movie browse and search against JSON movie data
  • Watchlist add/remove wired through the API
  • Vanilla frontend: fetch on load, forms, and DOM updates without a framework

What I Learned

This is where frontend and backend stopped being separate exercises. Keeping a JWT in localStorage, sending it on each request, and handling expired or missing tokens taught me why auth errors need clear status codes and UI feedback. JSON was fine for movie data here; I learned later (NovaBank) why a real database matters once you need relations and transactions.

WaveBox: music player

Why I Built It

After StreamFlix I wanted something with a Python backend. FastAPI made it quick to serve track data as JSON, and a plain HTML/CSS/JavaScript player on the front was enough to practise fetching structured data and driving playback controls from the DOM.

What I Built

  • FastAPI endpoints that return track and album data as JSON
  • Album grid and track list UI
  • Play, pause and track selection in vanilla JavaScript
  • Now-playing state updated from the API response

What I Learned

Small APIs still need a clear shape. Once the frontend expected a consistent JSON structure, changing a field name broke the player until I matched both sides. It was a good warm-up for treating the backend as the source of truth and the browser as something that only renders what it receives.

Algorithms & Data Structures (Year 2)

Why I Built It

Four timed C# exercises from the Algorithms & Data Structures module. Each repo came with starter code (menus, class shells, project structure). My job was to implement the core data structures and algorithms inside that framework.

What I Built

  • Emergency Call Queue System: custom queue ADT for calls by type and severity
  • Video Game Catalogue Manager: BST and AVL tree logic for search, update and removal
  • Aid Distribution Optimiser: QuickSort plus a greedy selector within a weight limit
  • Metro Network Route Planner: graph ADT with BFS, DFS and shortest-path routing

What I Learned

Working inside someone else's starter code is different from building from scratch. You have to read the given structure, implement only what the brief asks for, and still make the algorithms correct under a deadline. The ADS set was useful practice for that, and for seeing how queues, trees, sorting and graphs show up in small console apps.

Space Survival: Programming (Year 1)

Why I Built It

First year Programming assignment at Manchester Metropolitan University. Build a 2D game to a fixed spec in Processing (Space_Survival.pde). Your ship stays in the centre while obstacles fly in; you push the whole field away with the arrow keys and survive as long as you can.

What I Built

  • Menu, playing, and game-over states in a single Processing sketch
  • Obstacle spawning with difficulty scaling every 20 points
  • Circle-based collision detection and lives with shield invulnerability
  • Particle explosion effects when the ship is hit
  • Object-oriented structure: Player, Obstacle, Particle, and ExplosionAnimation classes

What I Learned

Game loops force you to think in frames: spawn, move, collide, draw, repeat. Splitting behaviour into classes made the sketch readable once it got past a few hundred lines. Collision and invulnerability frames were fiddly, getting hit once shouldn't drain every life in a single frame, so the shield timing mattered.
